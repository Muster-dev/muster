# Hooks

Hooks are executable shell scripts that perform the actual work of deploying, health-checking, rolling back, streaming logs, and cleaning up services. Each service gets its own hook directory.

## Hook Directory

```
your-project/.muster/hooks/<service>/
  deploy.sh      # build and deploy the service
  health.sh      # check if the service is healthy
  rollback.sh    # undo the last deploy
  logs.sh        # stream service logs
  cleanup.sh     # clean up resources (processes, containers)
```

All hooks are generated by `muster setup` from stack-specific templates. You can edit them freely after setup.

## Hook Types

| Hook | Purpose | Called By |
|------|---------|-----------|
| `deploy.sh` | Build and deploy the service | `muster deploy` |
| `health.sh` | Verify the service is running correctly | `muster deploy`, `muster status`, `muster dev` |
| `rollback.sh` | Revert to the previous version | `muster rollback`, deploy failure recovery |
| `logs.sh` | Stream or tail service logs | `muster logs <service>` |
| `cleanup.sh` | Stop processes, remove containers | `muster cleanup`, `muster dev` (on Ctrl+C) |

Exit codes:
- `0` -- success
- Non-zero -- failure (triggers recovery menu for deploy/rollback)

## Justfile Alternative

Hooks can also be defined as recipes in a `justfile` alongside or instead of shell scripts. If a `justfile` exists in the hook directory with a matching recipe name (`deploy`, `health`, `rollback`, `logs`, `cleanup`), muster uses it automatically. The justfile format is supported for both local and remote execution.

## Environment Variables

Hooks receive context via environment variables exported by muster before execution.

### General Variables

| Variable | Description |
|----------|-------------|
| `MUSTER_SERVICE_NAME` | Display name of the service (e.g. "API Server") |
| `MUSTER_DEPLOY_STATUS` | Deploy outcome: `success`, `failed`, `skipped` (post-deploy only) |
| `MUSTER_DEPLOY_TIMEOUT` | Timeout in seconds for the deploy (default: 120) |
| `MUSTER_DEPLOY_MODE` | Per-service deploy mode (e.g. `update`), if configured |
| `MUSTER_PROJECT_DIR` | Path to the project root |
| `MUSTER_CONFIG_FILE` | Path to `muster.json` |

### Kubernetes Variables

Exported when a service has `k8s` config in `muster.json`:

| Variable | Description | Example |
|----------|-------------|---------|
| `MUSTER_K8S_DEPLOYMENT` | Kubernetes deployment name | `myapp-api` |
| `MUSTER_K8S_NAMESPACE` | Kubernetes namespace | `production` |
| `MUSTER_K8S_SERVICE` | Service key with underscores converted to hyphens | `api-v1` |
| `MUSTER_DEPLOY_TIMEOUT` | Per-service timeout (if set in config) | `300` |
| `MUSTER_DEPLOY_MODE` | Per-service deploy mode (if set in config) | `update` |

The underscore-to-hyphen conversion for `MUSTER_K8S_SERVICE` handles the fact that Kubernetes does not allow underscores in resource names (e.g., service key `api_v1` becomes `api-v1`).

### Credential Variables

Exported when a service has `credentials.enabled: true` in `muster.json`:

| Variable | Description |
|----------|-------------|
| `MUSTER_CRED_<KEY>` | One per entry in `credentials.required[]`, uppercased |

Credential values are sourced according to the configured mode:

| Mode | Behavior |
|------|----------|
| `off` | No credentials loaded |
| `save` | Stored in OS keychain, prompted on first use |
| `session` | Cached in memory for the session, prompted on first use |
| `always` | Prompted every time |

Credentials are never stored in `muster.json` or committed to git.

### Git Variables

Exported when the project is inside a git repository:

| Variable | Description |
|----------|-------------|
| `MUSTER_GIT_COMMIT` | Current HEAD SHA after deploy |
| `MUSTER_GIT_PREV_COMMIT` | SHA of the last successful deploy |
| `MUSTER_GIT_COMMIT_COUNT` | Number of commits between prev and current |
| `MUSTER_GIT_DIFF_SUMMARY` | Short diff stat between prev and current |

### .env Auto-Loading

If a `.env` file exists in the project root, muster loads it before deploy/rollback/dev. Variables are exported as `KEY=VALUE` pairs without overriding existing environment variables. Cleanup happens automatically after the operation.

## Stack-Specific Templates

Setup generates hooks from templates in `templates/hooks/<stack>/`:

| Stack | Template Dir | Description |
|-------|-------------|-------------|
| `k8s` | `templates/hooks/k8s/` | `kubectl apply`, `kubectl rollout`, image tagging |
| `compose` | `templates/hooks/compose/` | `docker compose up`, `docker compose pull` |
| `docker` | `templates/hooks/docker/` | `docker build`, `docker run` |
| `bare` | `templates/hooks/bare/` | Generic shell commands, systemd restart |
| `dev` | `templates/hooks/dev/` | PID-managed background processes |

### Infrastructure Service Templates

Infrastructure services (redis, postgres, nginx, memcached, etc.) are auto-detected during setup and get specialized "pull-only" templates from `templates/hooks/<stack>/infra/`. These templates:

- Use `docker pull` / `docker compose pull` instead of `docker build`
- Skip source code compilation steps
- Use `kubectl rollout restart` instead of `kubectl set image` for K8s

This avoids building custom images for off-the-shelf infrastructure services.

### Dev Stack Templates

The `dev` stack uses two template variants:

- **App services** -- PID-managed background processes stored in `.muster/pids/`
- **Infra services** -- `docker compose` for databases, caches, etc.

The scanner auto-detects dev commands from project files:
- `package.json` -- `npm run dev`, `npm start`
- `go.mod` -- `go run .`
- `requirements.txt` -- Django/FastAPI/Flask dev servers

## Hook Security

Muster includes a security layer that validates hooks before execution.

### Dangerous Command Scanning

Before running a deploy hook, `_hook_security_check` scans the script for dangerous patterns:

| Pattern | Description |
|---------|-------------|
| `rm -rf /` | Filesystem destruction |
| `mkfs` | Disk format command |
| `dd if=` | Raw disk write |
| `curl \| bash` | Remote code execution |
| `wget \| sh` | Remote code execution |
| `chmod 777` | Insecure permissions |
| `nc -l` / `ncat -l` | Reverse shell listener |

If a dangerous pattern is detected, the hook is blocked and the user is warned.

### Manifest Integrity

Setup generates a signed manifest (`.muster/.hook_manifest`) containing SHA-256 hashes of all hook files. The manifest is HMAC-signed with a machine-local secret key (`~/.muster/.hook_signing_key`). Before each deploy, muster verifies:

1. The manifest signature is valid (not tampered with)
2. Hook file hashes match the manifest (files not modified by an attacker)

If verification fails, the deploy is blocked with a security warning.
